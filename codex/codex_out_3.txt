Project Purpose

This repo is a homework scaffold for building a small terminal coding agent called nanocli. The intended end state is a skill-aware CLI that uses the OpenAI Responses API for tool-calling, loads local SKILL.md files, exposes lightweight skill metadata implicitly, and injects full skill contents when the user explicitly prefixes a turn with $skill-name (hw3.md (line 21), hw3.md (line 64), core.py (line 83), cli.py (line 172)).

As implemented today, it is still a starter project. The CLI shell exists, the OpenAI request assembly exists, and the tool registry is defined, but several core behaviors are still NotImplemented, especially the skill discovery/injection logic and most non-bash tools (skill.py (line 16), skill.py (line 106), skill.py (line 121), tool.py (line 221), core.py (line 192)).

How To Run

The project is meant to be run with uv (README.md (line 7), pyproject.toml (line 15)).

Diagnosis:

The package itself is fine, and the wheel build includes the actual nanocli package. The failure is in the editable install behavior used by uv run here: the environment only has a nanocli.pth shim pointing at src/, but that path is not being applied to sys.path at startup, so the console scripts in .venv/bin/ try from nanocli... and fail with ModuleNotFoundError.

So the problem is not “the source code package is missing.” It is specifically:

uv run nanocli and uv run nanocli-mock depend on the editable install path shim
that shim exists at .venv/lib/python3.12/site-packages/nanocli.pth
but Python startup in this environment is not honoring it
once I manually append that exact path, import nanocli works immediately
In short: this is an editable-install / import-path initialization problem, not a missing module in src/ and not the HW TODO logic.

The separate test failures are still real and independent: even if the import-path issue were fixed, the tests would still fail because several functions are intentionally unimplemented.
