**Purpose**

This repository is a starter homework project for building a minimal tool-using command-line agent called nanocli. The assignment asks the student to upgrade a prior shell-only agent so it uses the OpenAI Responses API’s structured tool-calling flow instead of parsing bash blocks out of assistant text.

The clearest project description is in hw2.md and README.md. In practical terms, the intended behavior is:

1. Accept a user prompt in a terminal REPL.
2. Send conversation state to the OpenAI Responses API.
3. Let the model return structured tool calls such as bash, read_file, list_dir, and edit_file.
4. Execute those tools locally.
5. Send tool results back to the model as structured function_call_output items.
6. Repeat until the model returns a final assistant message.

The core architecture for that is in core.py, the terminal UI is in cli.py, and the tool registry and tool implementations are in tool.py.

One important caveat: this is intentionally incomplete starter code. The README and assignment both state that tests will initially fail until the TODOs in core.py and tool.py are implemented.

**How It Should Be Run**

Setup and standard commands are documented in README.md.

Primary workflow:

1. Install uv.
2. Install project dependencies through uv.
3. Run tests with:
   uv run pytest
4. Run lint/hooks with:
   prek run --all-files
5. Start the mock Responses API server with:
   uv run nanocli-mock
6. Start the CLI with:
   uv run nanocli --model nothing

The package entry points are declared in pyproject.toml:

- nanocli → cli.py
- nanocli-mock → mock_server.py

The CLI supports these notable flags in cli.py:

- --model
- --yolo
- --instructions-path
- --instructions-as-system
- --reasoning_effort

What they do:

- --model chooses the model name sent to OpenAI.
- --yolo skips approval prompts for non-read-only tools.
- --instructions-path loads the system prompt text, defaulting to tool_use_instructions.txt.
- --instructions-as-system sends instructions as an explicit system message instead of the instructions request field.
- --reasoning_effort passes a reasoning setting when supported.

**APIs Used**

The main external API is the OpenAI Responses API, via the Python SDK in core.py.

The key call is Session.stream_response_final, which uses:

- client.responses.create(...)
- tools=tool_specs()
- tool_choice="auto"
- stream=True
- store=False
- include=["reasoning.encrypted_content"]
- optional instructions or system message handling
- optional reasoning settings
- prompt_cache_key derived from the instructions text

The repo is built around these structured response item types from the Responses API:

- message
- reasoning
- function_call
- function_call_output

That mapping is explained both in hw2.md and in code in core.py.

There is also a local mock API server in mock_server.py. It mimics the streaming Responses API over HTTP/SSE so tests can verify the tool loop without calling the real service.

**Commands and Tools Used**

There are two layers of “commands” in this repo.

CLI commands exposed to the developer/user:

- uv run pytest
- uv run nanocli --model ...
- uv run nanocli-mock
- prek run --all-files

Agent tools exposed to the model via tool schemas in tool.py:

- bash
- read_file
- list_dir
- edit_file

What each tool does:

- bash: runs a shell command in the session working directory using subprocess.run with shell=True
- read_file: reads UTF-8 text files and returns numbered lines
- list_dir: lists directory contents in sorted order
- edit_file: performs exact-once search/replace edits in a UTF-8 text file

The tests in test_tool.py and test_core.py treat those tools as the formal behavior spec.

The mock server also supports trigger phrases for development/testing, such as mock:single, mock:multi, mock:read, and mock:edit, defined in mock_server.py.
