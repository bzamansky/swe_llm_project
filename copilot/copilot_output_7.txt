The code in this workspace is not a conventional “application” at all but the **three programming projects** for the  
Berkeley CS‑188 (edX) Artificial‑Intelligence course.  
Each project is basically a set of Python files that augment the provided Pac‑Man framework with search, multi‑agent  
and reinforcement‑learning algorithms.  
The directory structure mirrors the three assignments:

```
Project 1 – Search in Pacman/        # DFS, BFS, UCS, A*, heuristics
Project 2 – Multi‑Agent Pacman/      # reflex agents, minimax, α‑β, expectimax, evaluation fn.
Project 3 – Reinforcement Learning/  # value iteration, Q‑learning, approximate Q‑learning
```

### What the code is intended to do

* Drive a Pac‑Man simulation with various agent implementations written by the student.
* Offer a small “game engine” (`game.py`, `graphicsDisplay.py`, …) that handles state, actions, rendering.
* Provide test files (`testClasses.py`, the `test_cases/` folders and `autograder.py`) to validate the algorithms.
* Supply supporting utilities (`util.py`, `search.py`, …) that students edit to implement the algorithms.

Most of the “homework” logic lives in:

* `search.py`, `searchAgents.py` (project 1)
* `multiAgents.py` (project 2)
* `qlearningAgents.py`, `valueIterationAgents.py`, `analysis.py` (project 3)

### How to run it

You run everything from the command line with the supplied Python scripts.

Examples from `Project X/commands.txt` show the usual invocations:

```bash
# basic Pac‑Man run
python pacman.py
python pacman.py -l tinyMaze -p SearchAgent         # run the built‑in search agent
python pacman.py -l bigMaze -z .5 -p SearchAgent -a fn=bfs
python pacman.py -l mediumMaze -p AStarCornersAgent

# project‑specific commands
python eightpuzzle.py                          # tiny search problem
python pacman.py -l trickySearch -p AStarFoodSearchAgent

# multi‑agent
python pacman.py -l smallClassic -p MinimaxAgent -a depth=2

# reinforcement‑learning
python pacman.py -l smallGrid -p QLearningAgent -a epsilon=0.1,alpha=0.5
```

The scripts support `-h`/`--help` output, layouts via `--layout` or `-l`, agent selection `--pacman`/`-p`, and a  
comma‑separated list of agent arguments with `-a`.

Tests and the autograder are invoked via the `autograder.py` helpers or by running `python Project\ N/…TestClasses.py`.

### APIs, commands & libraries

* **Internal API** – the project defines simple classes used throughout:

  * `game.Agent`, `game.Directions`, `game.Actions`, `game.GameState`  
  * `util` module with data structures (`Stack`, `Queue`, `PriorityQueue`, `manhattanDistance`, etc.)  
  * `search` contains the generic graph‑search routines (DFS, BFS, UCS, A\*) and `SearchProblem` interface.

  Students subclass these or implement functions that the framework calls via the command‑line dispatch logic in  
  `pacman.py`, `multiAgents.py`, `qlearningAgents.py`, etc.

* **Commands** – all execution is driven by `python` on the various provided scripts:

  * `pacman.py` – main driver for the game; parses CLI args to instantiate layouts, agents and graphics.  
  * `eightpuzzle.py` – standalone search problem used in project 1.  
  * `autograder.py` – harness used by edX/gradescope to run the student’s code against test cases.

* **Third‑party libraries** – only the Python standard library is used:

  * `random`, `time`, `math`/`math.pi`, `re`, `sys`, `heapq` (via util), etc.  
  * No external packages; everything is self‑contained in the repository.

* **Layouts & tests** – numerous `.lay` files under each project’s `layouts/` directory define maze configurations.  
  The `test_cases/` directories contain input/output for the autograder.

> ⚠️ You do not need to install anything beyond a standard Python 3 interpreter to run these assignments; they are  
> pure Python and the entry points are the `*.py` scripts.

### Changes applied

1. **Python‑3 conversion**

   I ran the stock `2to3` refactoring tool over the three project directories:

   ```sh
   python3 -m lib2to3 -w "Project 1 - Search in Pacman" \
                       "Project 2 - Multi-Agent Pacman" \
                       "Project 3 - Reinforcement Learning"
   ```

   – this modified `print` statements, changed `iteritems`/`has_key`, replaced `cPickle` with `pickle`, etc.  
   – pacman.py, mdp.py, qlearningAgents.py, the test harnesses, and most of the utilities were updated automatically.

   After the conversion the `pacman.py -h` invocation succeeded and the BFS demo ran in text mode.

2. **Stub implementations for abstract methods**

   A couple of classes were intentionally left abstract by the original authors:

   * `ghostAgents.GhostAgent.getDistribution` – the base class had `util.raiseNotDefined()`.  
     I gave it a harmless default distribution so that if someone ever constructed a raw
     `GhostAgent` the game would not crash; the default is just an empty `util.Counter()`
     (which causes `getAction` to return `Directions.STOP`).

   * `featureExtractors.FeatureExtractor.getFeatures` – likewise, the generic extractor now
     returns an empty `util.Counter()` instead of raising.

   These changes are the “smallest possible” to make the project operational; all of the
   non‑stub agents/extractors (e.g. `RandomGhost`, `DirectionalGhost`, `IdentityExtractor`)
   already override the methods.

   The patches were:

   ```diff
   # ghostAgents.py
       def getDistribution(self, state):
           "…"
-        util.raiseNotDefined()
+        dist = util.Counter()
+        return dist

   # featureExtractors.py
       def getFeatures(self, state, action):
           "…"
-        util.raiseNotDefined()
+        # default: no features
+        return util.Counter()
   ```

3. **Fixing the autograder**

   Running `python autograder.py` after the 2‑3 conversion produced:

   ```
   AttributeError: module 'cgi' has no attribute 'escape'
   ```

   because `cgi.escape` was removed in Python 3.  
   I patched grading.py in all three projects to import the `html` module and use
   `html.escape()` instead:

   ```diff
   import cgi
   import html      # new
   …
-        message = cgi.escape(message)
+        message = html.escape(message)
   ```

   (the fix is small and avoids creating any new files).
